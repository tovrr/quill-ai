Title: Organize my project and repo folder

## Purpose

This prompt instructs an AI assistant (agent) to propose, plan, and produce a safe, reviewable repository reorganization. The agent will produce a recommended folder structure, a migration plan (file moves/renames), and an actionable patch (or list of git commands) for the user to review and apply.

## Inputs (provide when invoking)

- `scope` (string): `repo` (default) or a subpath such as `src/` or `docs/`.
- `style` (string): `conservative` (non-breaking, minimal moves) or `aggressive` (merge modules, canonicalize paths).
- `goals` (string[]): short bullets describing objectives (e.g., "consolidate integrations into src/lib/integrations", "add validator hooks").
- `constraints` (string[]): any hard constraints (e.g., "preserve history only via git mv is not required", "do not change production build outputs").
- `examples` (optional): reference repos or patterns to emulate.

## Agent Tasks (what the assistant must do)

1. Read repository metadata and `AGENTS.md`, `REPOSITORY_STRUCTURE_AUDIT.json`, and current top-level folders.
2. Produce a concise recommended folder layout (one paragraph + bullet list).
3. Generate a step-by-step migration plan, ordered by low-risk → high-risk actions.
4. For each step, list the exact git commands or an apply_patch-style diff that the maintainer can review and run.
5. Highlight any breaking changes, required env var changes, or CI updates.
6. Produce a small verification checklist and suggested tests to run after each migration step.

## Output format (strict)

The agent MUST return a JSON object inside fenced code block (or plain JSON when used programmatically) with these keys:

{
"summary": "short paragraph",
"recommendedStructure": [ { "path": "src/lib/integrations", "purpose": "external integrations" }, ... ],
"migrationPlan": [
{ "id": 1, "title": "Move integrations", "risk": "low", "commands": ["git mv src/lib/web-search.ts src/lib/integrations/web-search.ts"], "filesAffected": ["src/lib/web-search.ts"] },
...
],
"applyPatch": "optional string with git-friendly patch or apply_patch diff",
"checklist": ["npm run typecheck", "npm run test:smoke"],
"notes": "free-text with caveats and follow-ups"
}

## Hard rules for the agent

- Never modify production secrets, API keys, or credentials.
- Do not perform file deletion without explicit confirmation; prefer `git mv` or additions.
- Keep changes minimal for `conservative` style; for `aggressive`, propose smaller incremental steps rather than one large atomic move.
- Always include a rollback plan (git commands to revert) and a CI run checklist.

## Example invocations (use these as templates)

- Conservative:
  "Organize my project and repo folder", scope: "src/", style: "conservative", goals: ["move integrations into src/lib/integrations", "fix docs path references"]

- Aggressive:
  "Organize my project and repo folder", scope: "repo", style: "aggressive", goals: ["consolidate builders in src/lib/builder", "rename web-search.ts -> integrations/web-search.ts"], constraints: ["no runtime API changes" ]

## Clarifying questions (agent must ask if missing)

- Do you want git-level history preserved via `git mv` for every move? (yes/no)
- Are we allowed to change import paths across the repo, or do you prefer adding re-export shim files first?
- Which branch should migration changes be prepared on (e.g., `migration/reorg-YYYYMMDD`)?

## Follow-ups and extensions

- After generating the plan, offer to run static checks (typecheck, lint) and produce a patch file.
- Offer to create small smoke tests that validate runtime behavior after moves (e.g., run `npm run test:smoke`).

## Storage

Save this file at repository root as `.prompt.md` so local agents and Copilot/Editor tooling can discover and reuse the template.

End of `.prompt.md`
