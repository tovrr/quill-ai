## Executive Summary

Quill AI is a well-engineered personal AI agent and builder application built on Next.js 16, React 19, and Google Gemini. The codebase demonstrates strong security discipline (enforced CSP, HSTS, rate limiting, token-safe metrics), a clean modular architecture, and a thoughtful AI tier model. The product reached a genuine v1.0.0 milestone with a working builder, canvas preview, and multi-mode chat.

**However, the product is not yet commercially viable.** Three foundational blockers prevent revenue generation and customer trust: there is no payment processor, no social/OAuth login, and the sandbox execution capability is partially false-claimed. Additionally, observability is ephemeral (in-memory metrics reset on restart), CI/CD pipeline visibility is absent from the workspace, and external uptime monitoring is undeployed. The remaining risk surface is manageable with focused 4–8 week execution.

**Overall Maturity Rating: 6.5 / 10** — Strong technical foundation; pre-revenue; observability and monetization gaps are the principal risks.

## Domain 2 — Technical Architecture

### Stack Assessment

| Layer | Technology | Version | Risk |
|---|---|---|---|
| Framework | Next.js | 16.1.3 | Low — current major |
| UI runtime | React | 19.2.3 | Low — current |
| Language | TypeScript | (tsconfig strict) | Low |
| AI SDK | Vercel AI SDK | 6.0.141 | Low — current |
| Primary AI | Google Gemini (Flash-Lite / Flash / Pro) | API v1 | Medium — single provider dependency |
| Alt AI | OpenRouter (free model fallback) | — | Low |
| Auth | Better Auth | 1.5.6 | Low — recent version |
| ORM | Drizzle ORM | 0.45.2 | Low |
| Database | Neon Postgres (serverless) | @neondatabase/serverless 1.0.2 | Medium — vendor lock-in |
| Rate limiting | Upstash Redis + in-memory fallback | REST API | Low |
| Web search | Tavily | API key | Low |
| Code execution | Docker (local spawn) | python:3.13-slim | High — see A-03 |
| CSS | Tailwind CSS 4 | 4.2.2 | Low |
| Icons | Heroicons | 2.1.1 | Low |

### Findings

| ID | Severity | Finding |
|---|---|---|
| A-01 | High | **Single AI provider dependency.** Gemini is the sole provider for Think and Pro tiers. No fallback exists if Google's API degrades. The OpenRouter fallback is Fast-mode only and gated behind `OPENROUTER_FAST_ENABLED=true`. |
| A-02 | Medium | **In-memory metrics are volatile.** `src/lib/builder/metrics.ts` stores all route metrics in `globalThis`. A Vercel cold start or edge function lifecycle reset silently drops all historical data. No persistent metrics store (Prometheus, DataDog, Posthog, etc.) is connected. |
| A-03 | High | **Docker code executor has no production path.** `src/lib/docker-executor.ts` spawns a local Docker container via `child_process.spawn`. This works on developer machines but is incompatible with Vercel serverless deployments. `QUILL_SANDBOX_CONTAINER_ENABLED=true` is a deployment-time footgun — it will fail silently or crash in production. |
| A-04 | Medium | **React preview runs Babel standalone in the client iframe.** `src/components/agent/CanvasPanel.tsx` loads Babel in the browser for JSX transpilation. This adds ~250KB to the canvas interaction path and could be replaced by a server-side transpile step or a more targeted runtime. |
| A-05 | Low | **Neon Postgres is the sole datastore.** No read replica, no cache layer (Redis is used only for rate limiting). Complex queries against growing `message`/`chat` tables may degrade without index expansion. |
| A-06 | Low | **`src/proxy.ts` exists but appears unused by default.** Verify whether this is a development artifact or an intended production path. |

### Recommendations

- **A-R01 (Owner: Backend | Priority: High | Effort: 3d):** Add a thin provider abstraction that routes Think/Pro to a secondary provider (e.g., Anthropic Claude or OpenAI) when Gemini returns 429/503. `THINKING_MODEL_OVERRIDE` partially supports this but doesn't auto-failover.
- **A-R02 (Owner: Infra | Priority: Medium | Effort: 1d):** Replace in-memory metrics with a persistent sink. Minimum viable: push aggregated counters to the Neon DB on a write-behind schedule, or integrate a free tier of PostHog/Axiom.
- **A-R03 (Owner: Backend | Priority: High | Effort: 2d):** Either (a) remove `docker-executor` from the production build and gate it cleanly behind a feature flag with a build-time error if enabled on Vercel, or (b) replace with a cloud code execution service (e.g., E2B, Modal, fly.io microVM). This resolves both the false-claims risk and the production safety issue documented in TODOS.md.

## Domain 3 — Security

### Findings

| ID | Severity | Finding | Evidence |
|---|---|---|---|
| S-01 | Medium | **CSP `report-uri` endpoint (`/api/csp-report`) has no ingest, storage, or alerting.** Violations are reported but silently discarded if the endpoint does nothing. | `next.config.ts:37` |
| S-02 | Medium | **`script-src 'unsafe-inline'` is present in the enforced CSP (production).** This weakens XSS protection. Mitigating factor: no `'unsafe-eval'` in prod. Ideal path is CSP nonces via Next.js middleware. | `next.config.ts:25` |
| S-03 | High | **Entitlement bypass via env vars (`PAID_USER_EMAILS`, `PAID_USER_IDS`).** These are comma-separated plaintext lists in environment variables. Any operator with env access can silently grant or deny paid access to any user without an audit trail. Without a payment processor this is currently the only access gate. | `src/lib/auth/security.ts:35-44` |
| S-04 | Low | **`accounts` table stores `accessToken`, `refreshToken`, and `idToken` in plaintext.** For OAuth providers (once Google login is added), these are sensitive fields that should be encrypted at rest at the application layer, or the DB should enforce column encryption. | `src/db/schema.ts:50-68` |
| S-05 | Low | **`sessions` table stores `ipAddress` and `userAgent`.** This is PII under GDPR. No retention policy is documented or enforced. | `src/db/schema.ts:32-40` |
| S-06 | Low | **Tavily API key passed in the `body` of a POST request.** The standard Tavily API accepts `api_key` in the JSON body, but this means the key appears in request logs. Prefer the `Authorization: Bearer` header if the API supports it. | `src/lib/integrations/web-search.ts:52` |
| S-07 | Medium | **No `npm audit` gate in deployment or CI.** The `DEPLOYMENT_CHECKLIST.md` lists `npm install` but not `npm audit`. No CVE scanning policy is defined. | `TODOS.md:Audit-Driven Backlog` |
| S-08 | Low | **HSTS `max-age=31536000` is applied on all responses including HTTP.** The HSTS header should only be sent on HTTPS responses; on platforms like Vercel this is handled at the termination layer but could cause issues in custom proxy setups. | `next.config.ts:82` |

### Recommendations

- **S-R01 (Owner: Backend | Priority: High | Effort: 4h):** Implement the `/api/csp-report` endpoint to parse and log (or store) violation reports. Minimum: `console.error` with structured JSON; ideal: persist to DB or forward to a log aggregator.
- **S-R02 (Owner: Backend | Priority: Medium | Effort: 1d):** Replace entitlement env-var overrides with an audited admin UI (`/admin/model-usage` page already exists as a shell — expand its scope) once Stripe is implemented. Until then, document the trust model explicitly.
- **S-R03 (Owner: Backend | Priority: Medium | Effort: 3h):** Add `npm audit --audit-level=high` to the pre-deploy checklist and create a quarterly dependency upgrade issue template.
- **S-R04 (Owner: Backend | Priority: Low | Effort: 2h):** Add a CSP nonce via Next.js middleware to eliminate `'unsafe-inline'` from script-src in the enforced policy.

## Domain 4 — Data Management

### Schema Assessment

The data model is clean and minimal:
- `user`, `session`, `account`, `verification` — Better Auth standard tables
- `chat` — per-user conversation records with title, sharing flag, creation timestamps
- `message` — conversation messages with `partsJson` JSONB (added in migration 0002)
- `entitlement` — trial/paid plan tracking per user
- `modelUsage` — per-request AI cost records (tokens, cost, model, provider)

### Findings

| ID | Severity | Finding |
|---|---|---|
| D-01 | Medium | **No data retention policy.** Session records, messages, and model usage logs accumulate indefinitely. No scheduled cleanup job exists. |
| D-02 | High | **No GDPR/CCPA data deletion flow.** There is no "delete my account" endpoint or documented data erasure procedure. The `privacy/page.tsx` page exists but its content must match actual ability to fulfill deletion requests. |
| D-03 | Medium | **`partsJson` stores arbitrary `UIMessagePart[]` JSONB.** There is no schema validation on write; malformed or maliciously large payloads could corrupt the message history. Max size limits are not enforced at the DB layer. |
| D-04 | Low | **`modelUsage` table lacks a foreign key to `user`.** Cost attribution queries must join manually. Adding `userId` as an indexed foreign key would enable per-user cost dashboards. |
| D-05 | Low | **Chat sharing uses a boolean flag (`isPublic`) with no expiry or token.** Shared links never expire and cannot be revoked on a per-link basis — only by changing the flag. |

### Recommendations

- **D-R01 (Owner: Backend | Priority: High | Effort: 1d):** Implement a `/api/me/delete` endpoint that cascades deletion across `user`, `chat`, `message`, `session`, `entitlement`, `modelUsage`. Add a confirmation UI in settings.
- **D-R02 (Owner: Backend | Priority: Medium | Effort: 4h):** Add a Drizzle-scheduled cleanup query (or a cron webhook on Vercel) to purge sessions expired > 30 days and model usage records > 6 months.
- **D-R03 (Owner: Backend | Priority: Medium | Effort: 2h):** Add a `CHECK` constraint or Zod validation before `saveMessage()` that enforces a maximum `partsJson` size.
