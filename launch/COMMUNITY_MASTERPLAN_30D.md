# Community Masterplan - 30 Days

## Goal

Build an initial public community around Quill AI with repeatable feedback loops and contributor onboarding.

Primary target by day 30:
- 100+ GitHub stars
- 10+ active discussion participants
- 3+ external issue reporters
- 1+ external merged contribution

## Guiding Rules

- Focus on proof before promotion: demo first, then distribution.
- Keep a weekly shipping rhythm.
- End every public post with one concrete question.
- Track outcomes every week and adapt quickly.

## Week 1 - Foundation and Proof

Objective: create trust assets that convert visitors into testers.

Tasks:
1. Replace demo placeholder with real GIF (12 to 20 seconds).
2. Create two static screenshots (agent flow and canvas preview).
3. Enable GitHub Discussions.
4. Pin three discussion threads:
   - Show and tell
   - Feature requests
   - Bugs and fixes
5. Publish GitHub release v1.0.0 and one launch post on X and LinkedIn.

Acceptance:
- Demo GIF is live in README.
- Discussions are enabled and pinned.
- At least 5 qualitative comments across channels.

## Week 2 - Audience Seeding

Objective: create recurring inbound traffic from founder and builder communities.

Tasks:
1. Publish 2 X posts and 1 LinkedIn post with different angles.
2. Share in 3 relevant communities with contextual intro:
   - Indie Hackers
   - Reddit (relevant subreddits)
   - Dev founder or AI builder Discord/Slack
3. Open 5 labeled issues for contributors:
   - good first issue
   - help wanted
4. Add acceptance criteria to each issue.

Acceptance:
- 3 community shares completed.
- 5 contributor-friendly issues published.
- At least 10 new stars week over week.

## Week 3 - Contributor Activation

Objective: convert observers into contributors.

Tasks:
1. Publish a short "How to contribute" post linking to CONTRIBUTING and issue labels.
2. Host one async feedback thread in Discussions with product roadmap options.
3. Prioritize and respond to all external issues within 24 hours.
4. Convert top repeated feedback into one public milestone.

Acceptance:
- Response SLA of under 24h for new external issues.
- One roadmap milestone visible and linked from discussions.
- At least one external PR opened.

## Week 4 - Retention and Cadence

Objective: establish visible consistency and keep contributors engaged.

Tasks:
1. Ship a small iterative release with clear changelog.
2. Publish a "What changed in 30 days" summary post.
3. Highlight community contributors publicly.
4. Document next 30-day roadmap based on feedback data.

Acceptance:
- One follow-up release published.
- One retrospective post published.
- Next 30-day plan committed in repository.

## KPI Dashboard (Weekly)

Track these metrics every Monday:
- GitHub stars delta
- Unique issue reporters
- Discussion participants
- External PRs opened and merged
- Median maintainer first response time

## Risk Controls

- If no traction after week 2:
  - Reduce channels and focus on one where responses are highest.
  - Tighten messaging around one core use case only.
- If issues exceed capacity:
  - Freeze feature intake for one week and focus on triage.
- If contributor conversion is low:
  - Reduce issue complexity and improve reproduction steps.

## Execution Note

This plan is intentionally post-critical-blockers. Execute after business-critical reliability and billing blockers are stabilized.
